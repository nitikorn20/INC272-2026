# Final Project Guide

## Project Goal

Design and implement a small browser-based monitoring or control application that works with the course simulator or another approved data source.

* * *

## Suitable Project Types

- monitoring dashboard
- browser-based control panel
- status and alert interface
- mixed monitoring and control interface
- small device-oriented web frontend

* * *

## Scope Guidance

The project should be small enough to finish within the onsite project period.

### Good Scope

- one clear problem
- one focused dashboard or a small number of views
- a working monitoring or control flow
- a demo that can be explained clearly

### Bad Scope

- too many screens
- backend complexity unrelated to the course goal
- infrastructure work that takes attention away from the browser application
- a project that cannot be demonstrated clearly in Week 06

* * *

## Expected Features

Projects should normally include:

- a clear interface layout
- at least one monitoring element
- at least one control element
- at least two simulator features (e.g. LED + ADC, PSW + LED)
- visible feedback for user actions
- readable code organization

* * *

## Submission Format

Each group submits a project folder containing:

- `index.html` — the main application page
- `main.js` — the application logic
- `README.md` — project documentation (see template below)
- any additional files used by the project (CSS, images, etc.)

The `README.md` must be written in Markdown. A template is provided at [`docs/project-readme-template.md`](./project-readme-template.md). Copy it into the project folder, fill in every section, and remove the instruction line at the top.

### GitHub Bonus

Groups that push their project to a public GitHub repository receive bonus points.

- The repository must be **public** so the instructor can access it without login.
- Submit the repository URL via **LEB2 during Week 06**.
- The instructor will review the latest state of the repository at the time of submission. Commit history is not evaluated.

To learn how to push a project to GitHub, watch the tutorial linked in the Week 06 README.

## Suggested Internal Rubric

The final project is one main assessment category worth `70%` of the course grade. A practical internal rubric is:

| Area | Suggested Weight |
|------|------------------|
| fit to the brief and scope clarity | `10%` |
| technical correctness | `25%` |
| interface design and usability | `15%` |
| code and repository quality | `10%` |
| checkpoint progress | `10%` |
| final demo and presentation | `30%` |

* * *

## Evaluation Questions

### Fit to the Brief

- Is the project goal clear?
- Is the chosen scope realistic for the course timeline?

### Technical Correctness

- Does the interface work correctly?
- Do monitoring values and control actions behave as expected?

### Interface Quality

- Is the interface readable and usable?
- Is feedback clear and well placed?

### Code and Repository Quality

- Is the project structure manageable?
- Are file organization and naming clear enough to follow?

### Demo Quality

- Can the team explain the workflow clearly?
- Does the demo show the real value of the project?

* * *

## Final Reminder

Keep the project focused. A smaller project with a clear and reliable demo is better than an over-scoped project with incomplete features.
